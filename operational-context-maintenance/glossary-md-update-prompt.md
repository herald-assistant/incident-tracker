# glossary.md update prompt

You maintain `src/main/resources/operational-context/glossary.md`.

Treat the attached operational-context YAML and MD files as the catalog write
model. Use the current glossary structure and headings already present in this
catalog.

## Write/read model boundary

`glossary.md` is a write-model source file for local language, not the FE/LLM
read model. Do not duplicate system, process, integration, repository or team
facts here just to make term entries look complete. Complete views for FE, LLM
and tools are generated by the application read model: relation index,
code-search, implementation, flow and blast-radius projections.

Never add manual backlinks. A term may mention canonical related ids when the
relationship is part of the term meaning, but do not maintain broad lists of
every system, process or integration that happens to use the term.

Typed references and free-text hints must stay separate:

- Use `Canonical references` only for typed operational-context refs such as
  `system:<id>`, `process:<id>`, `bounded-context:<id>`, `integration:<id>`,
  `repository:<id>`, `team:<id>`, `term:<id>` or `handoff-rule:<id>`.
- Use `Related terms` for glossary term ids or term-level language links.
- Put human-readable aliases, explanatory phrases and non-catalog names in
  `Aliases`, `Not to confuse with`, `Responsibility hints`, `LLM tool hints` or
  `Notes`, not in typed reference lists.
- Do not invent a typed reference when the target id cannot be verified in the
  attached catalog; add a gap or explanatory hint instead.

## Fact ownership for this file

Keep facts here only when they describe local language:

- domain terms, aliases, acronyms and stable operator vocabulary,
- short definitions in the local system language,
- common confusion points and "do not confuse with" notes,
- durable match signals for terms,
- term-level gaps when a definition or usage boundary is unclear.

Do not maintain these facts here:

- system runtime identity that belongs to `systems.yml`,
- repository source layout or code-search strategy that belongs to
  `repo-map.yml`,
- integration participants and transport contracts that belong to
  `integrations.yml`,
- process boundaries or ordered steps that belong to `processes.yml`,
- ownership or routing that belongs to `teams.yml` or `handoff-rules.md`.

## Update rules

- Prefer editing an existing term when evidence matches a name, alias, acronym,
  marker, domain phrase or documented definition.
- Add a new term only when the evidence identifies vocabulary that will recur
  in code, logs, business analysis or operator handoff.
- Keep definitions short and factual. Do not paste stack traces, full payloads,
  secrets, transient URLs or long code excerpts.
- Use stable aliases and match signals only when they help AI resolve evidence
  to a concept.
- If evidence suggests a relation whose owner is another file, update that file
  with its own prompt or add an actionable glossary gap instead of duplicating
  the fact here.
- Validate referenced ids against the attached YAML/MD files, but do not add
  reciprocal references only for symmetry.
- Every `Canonical references` value must resolve to an existing typed catalog
  id. Every `Related terms` value must resolve to an existing glossary term id
  unless it is deliberately written as explanatory prose outside the list.

## Output rules

Return the complete updated `glossary.md` content only.
Do not include commentary, markdown fences, diffs, or explanations.
