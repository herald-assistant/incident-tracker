## Write/read model boundary

`handoff-rules.md` is a write-model source file for operator-facing routing
policy, not the FE/LLM read model. Do not duplicate system, process,
integration, repository or team facts here just to make a rule look like a full
entity envelope. Complete views for FE, LLM and tools are generated by the
application read model: relation index, code-search, implementation, flow and
blast-radius projections.

Never add manual backlinks. A rule may link to the entities it explicitly
governs, but do not maintain broad reverse lists of every entity that references
the rule.

Typed references and free-text routing labels must stay separate:

- `Route to` is an operator-facing routing decision or label, not a typed team
  relation.
- `Partner teams` is an operator-facing collaboration hint, not canonical team
  ownership.
- Put catalog entity links in `Operational context links` as typed refs, e.g.
  `system:<id>`, `process:<id>`, `bounded-context:<id>`, `integration:<id>`,
  `repository:<id>`, `team:<id>`, `term:<id>` or `handoff-rule:<id>`.
- `Operational context links` is the canonical owner for process-handoff
  applicability. Do not ask `processes.yml` to add reciprocal
  `references.handoffRules` for the same rule.
- Do not put unverifiable human team names, vendor labels or route descriptions
  into typed reference lists; keep them as route/partner labels or add a gap.

## Fact ownership for this file

Keep facts here only when they describe practical handoff behavior:

- when to route, escalate or involve a partner team,
- required evidence before handoff,
- expected first action for the receiving side,
- routing constraints, ownership caveats and collaboration hints,
- rule-level gaps when routing policy is unclear.

Do not maintain these facts here:

- team responsibility records that belong to `teams.yml`,
- system runtime identity that belongs to `systems.yml`,
- repository source layout or code-search strategy that belongs to
  `repo-map.yml`,
- integration participants and transport contracts that belong to
  `integrations.yml`,
- process boundaries or ordered steps that belong to `processes.yml`.

## Update rules

- Prefer editing an existing rule when evidence matches route target, symptom,
  required evidence, partner team or escalation condition.
- Add a new rule only when the evidence identifies repeatable routing behavior
  that an operator can apply again.
- Keep rules practical and short. Do not paste full incident narratives,
  stack traces, secrets, transient URLs or long payloads.
- Use `requiredEvidence`-style checklists when the current file structure
  supports them.
- If evidence suggests ownership rather than handoff policy, update
  `teams.yml` with its own prompt or add an actionable gap instead of
  duplicating the fact here.
- Validate referenced ids against the attached YAML/MD files, but do not add
  reciprocal references only for symmetry.

## Output rules

Return the complete updated `handoff-rules.md` content only.
Do not include commentary, markdown fences, diffs, or explanations.
